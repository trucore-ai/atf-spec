# ATF Receipt Specification

This document defines the **ATF receipt** format and the deterministic rules used to compute and verify its `content_hash`.

The receipt is intended to be:
- **machine-parseable**
- **deterministically verifiable**
- **tamper-evident**
- suitable for logging, audits, and downstream trust proofs

## Goals

A valid receipt should allow an independent verifier to answer:

1. What request was evaluated? (via stable identifiers and hashed inputs)
2. What decision was produced? (`allow` / `deny` / `error`)
3. Was the receipt altered after issuance? (via `content_hash`)

## Non-goals

- Proving correctness of *internal* policy reasoning beyond what is included in receipt fields.
- Replacing cryptographic signatures (receipts can be signed separately; signing is out of scope here unless explicitly added).
- Documenting internal enforcement logic, policy taxonomies, or action schemas.

## Receipt object

### Format

- The receipt is a **single JSON object**.
- All keys are **lower_snake_case**.
- The receipt MUST include `receipt_version` and `content_hash`.

### Required fields

| Field | Type | Description |
|---|---|---|
| `receipt_version` | string | Semantic version of this receipt format (e.g., `"1.0"`). |
| `issued_at` | string | RFC 3339 timestamp in UTC (e.g., `"2026-02-28T20:15:31Z"`). |
| `request_id` | string | Correlation ID passed through from caller if provided; otherwise generated by ATF. Must be stable for the request lifecycle. |
| `engine` | object | Identifies the receipt-producing engine. |
| `decision` | object | High-level decision outcome. |
| `inputs` | object | Hashes/identifiers describing the evaluated request. |
| `policy` | object | Identifies the policy configuration used. |
| `content_hash` | string | Deterministic SHA-256 hex of the canonicalized receipt content (excluding `content_hash` itself). |

### Required nested fields

#### `engine`

| Field | Type | Description |
|---|---|---|
| `engine.name` | string | Engine identifier (e.g., `"atf"`). |
| `engine.build` | string | Build or release identifier for traceability. |
| `engine.api_version` | string | API contract version used to emit this receipt. |

#### `decision`

| Field | Type | Description |
|---|---|---|
| `decision.status` | string | One of: `"allow"`, `"deny"`, `"error"`. |
| `decision.reason_code` | string | Stable, opaque code for downstream automation. Values are engine-defined and not enumerated in this spec. |
| `decision.summary` | string | Human-readable summary (short). |
| `decision.deterministic` | boolean | MUST be `true` if verification should succeed deterministically. |

#### `inputs`

`inputs` describes the evaluated request in privacy-preserving form.

| Field | Type | Description |
|---|---|---|
| `inputs.content_type` | string | MIME type of the original request body (e.g., `"application/json"`). |
| `inputs.request_hash` | string | SHA-256 hex of the canonicalized request body used for evaluation. |
| `inputs.context_hash` | string | SHA-256 hex of canonicalized evaluation context (may be hash of `{}` if none). |

> Note: Canonicalization uses JSON Canonicalization Scheme (JCS, RFC 8785). See below and `spec/verification.md`.

#### `policy`

| Field | Type | Description |
|---|---|---|
| `policy.policy_hash` | string | Stable hash identifying the policy configuration used. |
| `policy.policy_version` | string | Policy format version string (not the same as `receipt_version`). |

### Optional fields

The receipt MAY include additional engine-defined fields (e.g., warnings, trace identifiers, error details). Optional fields are included in the `content_hash` computation if present, but their schemas are engine-defined and not specified here.

## Deterministic `content_hash`

### Canonicalization rules

To compute `content_hash`, the receipt MUST be canonicalized deterministically.

This spec uses **JSON Canonicalization Scheme (JCS, RFC 8785)** semantics:

- Object keys sorted lexicographically (UTF-16 code unit order per RFC 8785)
- No insignificant whitespace
- Numbers normalized per RFC 8785 rules
- Strings encoded in UTF-8
- Arrays preserve order

### Hash computation

1. Take the receipt JSON object.
2. Remove the `content_hash` field entirely.
3. Canonicalize the remaining JSON via JCS (RFC 8785).
4. Compute SHA-256 over the UTF-8 bytes of the canonical JSON.
5. Encode as lowercase hex string.
6. Set `content_hash` to this value.

### Example pseudocode

```python
canonical = JCS(receipt_without_content_hash)
content_hash = sha256(canonical.encode("utf-8")).hexdigest()
```

### Determinism guarantees

If `decision.deterministic` is `true`, then:

- Recomputing `content_hash` using the above method MUST match the `content_hash` field.
- `inputs.request_hash` and `inputs.context_hash` MUST be computed using the same canonicalization method described in `spec/verification.md`.

## What the receipt does not guarantee

A valid `content_hash` guarantees only:

- The receipt content has not been altered since issuance (tamper-evidence).

It does **NOT** prove:

- Who issued it (unless separately signed).
- That the engine's internal reasoning was "correct."
- That policy enforcement was bug-free.
- That the request was executed (that requires execution proofs outside this spec).

Those properties can be layered later (signatures, attestations, execution receipts) while keeping this deterministic core stable.
